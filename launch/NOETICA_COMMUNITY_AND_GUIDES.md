# Noetica Community & User Guides

*This document contains the community management and user onboarding materials (Deliverables 15–20) for the Noetica public launch.*

***

## 15. Community Guidelines

**Welcome to the Noetica Community.**

This space exists to facilitate the development of an open-source scientific intelligence network. To maintain a productive, rigorous, and respectful environment, all participants must adhere to the following principles:

1. **Scientific Rigor First:** Discussions should be rooted in evidence, architecture, and structural biology. Avoid speculation, hype, or unsupported claims. 
2. **Constructive Criticism:** I expect you to break the system and point out its flaws. When you do, provide context, logs, and potential solutions. "This doesn't work" is not helpful. "The ingestion pipeline fails on PDF parsing at line 42 due to a timeout" is helpful.
3. **Transparency:** If you are testing Noetica for a specific commercial or academic project, disclose it if relevant to your feature requests.
4. **No Marketing or Hype:** Do not use this space to promote unrelated startups, products, or tokens. Do not exaggerate Noetica's capabilities. 
5. **Respect the Code of Conduct:** Harassment, discrimination, and toxic behavior will result in immediate removal. We are here to build infrastructure, not egos.

## 16. Beta Tester Welcome Guide

**Welcome to the Noetica Closed Beta (v0.1.0).**

Thank you for being one of the first 500 users to test Noetica. 

**What you need to know:**
- **This is an experimental build.** You are testing the foundational architecture. Expect bugs, UI glitches, and unoptimized queries.
- **The goal is validation.** I need to know if the BioSignal scoring algorithms accurately reflect the structural integrity of the literature you input.
- **Your feedback shapes the roadmap.** Because the user base is strictly limited, your bug reports and feature requests will have a direct impact on the v0.2.0 architecture.

**Your Objectives:**
1. Upload a research paper or dataset you are deeply familiar with.
2. Review the causal connections and scores generated by the engine.
3. Identify false positives, hallucinated connections, or missed structural links.
4. Report your findings via the GitHub Issues tab.

I appreciate your time and your commitment to open scientific infrastructure.

## 17. First Login Guide

**Getting Started with Noetica**

1. **Authentication:** 
   Navigate to the portal and authenticate using your GitHub or Google credentials. Ensure you are using the email address approved for the closed beta.

2. **The Dashboard:**
   Upon login, you will see the Discovery Timeline. Currently, it is populated with a baseline dataset. 

3. **Running Your First Query:**
   - Navigate to the **Ingestion** tab.
   - Upload a standard academic PDF or input a DOI.
   - The engine will begin processing. Note: In this beta, processing may take up to 3 minutes depending on server load.

4. **Reviewing the Output:**
   - Once processed, navigate to the **Analysis** tab.
   - Review the 8 Core Questions checklist.
   - Examine the BioSignal score and the generated knowledge graph.

5. **Providing Feedback:**
   Click the "Flag Artifact" button on any node or score that appears structurally incorrect. This will log the error for architectural review.

## 18. Bug Reporting Guide

**How to Report a Bug in Noetica**

Because Noetica is an open-source project, all bug tracking is handled publicly via GitHub Issues. 

When submitting a bug report, use the `[BUG]` template and include the following:

1. **System Environment:** Browser, OS, and whether you are running a local instance or using the cloud beta.
2. **Steps to Reproduce:** A step-by-step list of exactly what you did before the failure occurred. 
3. **Expected vs. Actual Output:** What you thought the engine would do versus what it actually did.
4. **Logs/Error Codes:** Paste any terminal errors, browser console errors, or failed query IDs.
5. **Data Source:** If the engine failed during ingestion, please link the specific DOI or paper (if public) that caused the crash.

*Note: Do not report slow load times as bugs unless a query times out completely. Optimization is a known issue for v0.2.0.*

## 19. Feature Request Guide

**How to Request a Feature**

I want Noetica to be shaped by the researchers who use it. However, to keep the architecture focused, feature requests must align with the core mission: prioritizing evidence over popularity.

To request a feature, open a GitHub Discussion under the `Ideas` category and provide:

1. **The Problem:** What specific friction are you experiencing in your research workflow?
2. **The Proposed Solution:** How should Noetica solve this? (e.g., "Add an export-to-CSV function for the knowledge graph nodes.")
3. **The Scientific Value:** How does this feature improve the structural validation of discoveries?

Features that request "trending papers" or "citation counting" will be rejected, as they fundamentally conflict with Noetica's design philosophy.

## 20. Contributing Guide (Summary)

**Contributing to Noetica**

Noetica is completely open-source. Whether you are a bioinformatician optimizing the scoring algorithms or a frontend developer fixing UI inconsistencies, your contributions are welcome.

**How to Contribute:**
1. **Fork the Repository:** Clone the `main` branch to your local environment.
2. **Find an Issue:** Look for issues tagged `good first issue` or `help wanted`.
3. **Branch Naming:** Use `feature/your-feature` or `bugfix/issue-number`.
4. **Pull Requests:** Ensure all local tests pass before submitting a PR. Include a detailed description of your architectural changes.
5. **Code Review:** I review all PRs manually. Expect technical feedback before a merge.

For the full architectural breakdown, please read the `ARCHITECTURE.md` file in the repository root before making structural changes to the codebase.
